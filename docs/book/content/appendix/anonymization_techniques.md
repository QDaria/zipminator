# Appendix: Anonymization Techniques

This appendix provides a detailed explanation of the algorithms and trade-offs behind each of Zipminator's 10 anonymization levels.

## Level 1: SHA-256 Hashing

**Algorithm**: Apply SHA-256 to the string representation of each value.

**Properties**:
- One-way: cannot recover the original value from the hash
- Deterministic: same input always produces the same hash (enables deduplication)
- Collision-resistant: computationally infeasible to find two inputs with the same hash

**Trade-offs**:
- Vulnerable to rainbow table attacks on low-entropy inputs (e.g., short names)
- Does not support range queries or aggregation
- No salt: identical values across datasets produce identical hashes (linkage risk)

**When to use**: Internal deduplication, audit trail integrity checks, simple pseudonymization where linkage risk is acceptable.

## Level 2: Quantum-Random Replacement

**Algorithm**: Replace each value with a 10-character string of uppercase letters and digits, generated by the `QuantumRandom` engine.

**Properties**:
- Non-deterministic: same input produces different outputs on each run
- When quantum entropy is available, randomness is seeded from IBM Quantum hardware
- Falls back to `os.urandom` when quantum entropy is not available

**Trade-offs**:
- Destroys all original information (not reversible)
- Breaks referential integrity (same person gets different random strings in different runs)
- No utility for analysis

**When to use**: Complete de-identification where no analytical utility is needed. Data destruction that preserves column structure.

## Level 3: Deterministic Tokenization

**Algorithm**: Map each unique value to a token using `TOKEN_{MD5(value)[:8].upper()}`. The mapping is stored in `engine._token_maps[column_name]`.

**Properties**:
- Deterministic: same input always produces the same token
- Reversible: the token map can be used to recover original values (if retained)
- Preserves referential integrity across columns and datasets

**Trade-offs**:
- The token map is a sensitive asset; if compromised, all tokenization is reversed
- MD5 truncation to 8 hex chars creates a collision space of ~4.3 billion; sufficient for most datasets but not guaranteed unique for very large datasets
- Vulnerable to known-plaintext attacks if an adversary knows some original values

**When to use**: Pseudonymization for systems that need consistent identifiers (e.g., joining tokenized datasets). PCI-DSS tokenization of payment card numbers.

## Level 4: Generalization

**Algorithm**:
- Numeric values: bucket into ranges of configurable size (default: 10). `value -> "{lower}-{upper}"`
- Text values: reduce to first-letter category. `"Alice" -> "CATEGORY_A"`

**Properties**:
- Reduces precision while preserving coarse patterns
- Non-reversible (many-to-one mapping)

**Trade-offs**:
- Bucket size directly controls the privacy/utility trade-off
- Small buckets: more utility, less privacy
- Large buckets: more privacy, less utility
- Text generalization is very coarse; consider L7 (synthetic) for better utility

**When to use**: Statistical analysis where exact values are not needed. Age ranges, salary bands, geographic regions.

## Level 5: Suppression

**Algorithm**: Replace all values in the column with `pd.NA` (pandas null).

**Properties**:
- Complete removal of data
- Column structure preserved (schema compatibility)

**Trade-offs**:
- Zero analytical utility for the suppressed column
- The column's existence may still leak information (presence of a "diagnosis" column implies medical context)

**When to use**: GDPR right to erasure. Columns that must exist in the schema but cannot contain any values. Removing quasi-identifiers that cannot be safely generalized.

## Level 6: Quantum Noise Injection

**Algorithm**:
- Numeric: add noise proportional to the value. `value + (value * 0.1 * uniform(-1, 1))`, where the uniform random number comes from `QuantumRandom`
- Non-numeric: falls back to quantum-random replacement (same as L2)

**Properties**:
- Preserves approximate magnitude and distribution shape
- Noise is multiplicative (10% of value), so larger values get larger absolute noise

**Trade-offs**:
- No formal privacy guarantee (unlike L9 differential privacy)
- Aggregate statistics (mean, variance) are approximately preserved
- Individual values can shift by up to 10%, which may be insufficient for high-sensitivity data

**When to use**: Approximate data sharing where exact values are sensitive but trends matter. Preliminary data exploration before applying stronger anonymization.

## Level 7: Synthetic Data Generation

**Algorithm**: Use the Faker library to generate realistic synthetic values based on column name hints. Recognized column names: name, email, phone, address, company, ssn, credit card.

**Properties**:
- Output looks realistic but has no connection to original data
- Column-name-aware: a "name" column gets fake names, an "email" column gets fake emails
- Falls back to random 12-character strings if Faker is not installed

**Trade-offs**:
- No statistical relationship between synthetic and original data
- Cannot be used for analysis that depends on actual distributions
- Faker data may contain culturally biased defaults (US-centric names, addresses)

**When to use**: Demo datasets, testing environments, UI mockups, training data for ML pipelines where privacy is paramount but realistic-looking data is needed.

## Level 8: k-Anonymity Grouping

**Algorithm**: Generalize quasi-identifiers until every combination of values appears at least k times in the dataset. Default k=5. Numeric columns are bucketed (same as L4); categorical columns are generalized.

**Mathematical foundation**: A dataset satisfies k-anonymity if every record is indistinguishable from at least k-1 other records with respect to the quasi-identifiers.

**Properties**:
- Formal privacy property: prevents singling out individuals
- Preserves more utility than suppression (L5) or synthetic data (L7)

**Trade-offs**:
- Vulnerable to homogeneity attacks (all k records in a group share the same sensitive value)
- Vulnerable to background knowledge attacks (adversary knows additional information)
- Higher k provides more privacy but reduces data utility
- Small datasets may require aggressive generalization to achieve k=5

**When to use**: Research datasets shared with external parties. HIPAA Safe Harbor de-identification. Any scenario where formal group-based privacy is required.

## Level 9: Differential Privacy (Laplace Noise)

**Algorithm**:
- Numeric: add Laplace noise calibrated to the data sensitivity. `value + Laplace(0, sensitivity/epsilon)` where `sensitivity = max - min` and `epsilon = 1.0`
- Categorical: randomized response with probability p=0.75 (report true value with 75% probability; random value with 25%)

**Mathematical foundation**: A mechanism M satisfies epsilon-differential privacy if for any two neighboring datasets D and D' (differing in one record), and any output set S: `Pr[M(D) in S] <= e^epsilon * Pr[M(D') in S]`

**Properties**:
- Formal, composable privacy guarantee
- Robust against arbitrary background knowledge
- The privacy budget (epsilon) controls the privacy/utility trade-off

**Trade-offs**:
- Epsilon=1.0 is a common default, but the optimal value depends on the use case
- Smaller epsilon = more privacy, more noise, less utility
- Larger epsilon = less privacy, less noise, more utility
- Laplace mechanism is optimal for numeric queries; randomized response is used for categorical data
- Requires numpy

**When to use**: Publishing statistical datasets, government open data, academic research, any scenario requiring formal mathematical privacy guarantees.

## Level 10: Paillier Homomorphic Encryption

**Algorithm**: Encrypt numeric values using the Paillier cryptosystem. The encrypted values support two operations without decryption:
- **Addition**: `Enc(a) * Enc(b) = Enc(a + b)`
- **Scalar multiplication**: `Enc(a)^k = Enc(a * k)`

**Mathematical foundation**: Paillier encryption is based on the decisional composite residuosity assumption. Security relies on the hardness of factoring large composites (similar to RSA).

**Properties**:
- Encrypted values can be summed and averaged without decryption
- The private key holder can decrypt at any time
- Semantically secure (same plaintext encrypts to different ciphertexts)

**Trade-offs**:
- Computationally expensive (orders of magnitude slower than symmetric encryption)
- Only supports addition and scalar multiplication (not general computation)
- Paillier is not post-quantum secure (vulnerable to Shor's algorithm); for quantum resistance, use Kyber768 for key exchange and AES-256-GCM for bulk encryption
- Requires the `phe` library; falls back to L9 if not installed
- Non-numeric columns fall back to SHA-256-based encryption

**When to use**: Secure multi-party computation, privacy-preserving machine learning (federated learning), scenarios where a data processor must compute on encrypted data without seeing the plaintext.
